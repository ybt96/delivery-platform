# 地方特色局域网络物资配送平台 - 需求分析与系统架构设计文档

## 一、 引言与项目背景

本项目致力于打造一个服务于特定地域（如老家乡镇、村落、社区）的综合性本地生活服务平台。该平台有别于传统的全国性电商，其核心竞争力在于**“地缘信任”**、**“信息透明”**以及**“互助共享”**。

平台旨在解决下沉市场（乡镇级）以下几个痛点：
1. 本地商品信息极度不对称（买不到急需配件，不知道哪家菜最新鲜）。
2. 缺乏灵活的最后三公里配送服务。
3. 本地实体商家缺乏有效的联合促销手段，单打独斗。
4. 乡镇出行不便，尤其是老人就医、儿童上下学等刚需场景，缺乏安全可靠的顺路搭乘渠道。

## 二、 需求分析

### 2.1 核心功能需求 (Functional Requirements)

#### 1. 地方百货信息平台 (商品/服务导览模块)
* **分类导航：** 涵盖新鲜蔬菜、美食餐馆、五金建材、日用百货、交通工具（自行车/电动车）等本地高频刚需品类。
* **信息透明化展示：** 不强制在线交易，但必须展示“哪里有货”、“货品实拍/评价（好赖）”、“参考价格区间”。
* **商家主页与库存概览：** 商家可自主更新今日特价、新鲜到货信息，用户可直观看到本地商家的主打产品。

#### 2. 预约配送上门服务 (履约模块)
* **灵活预约：** 支持按需选择配送时间段（如：上午赶集后、下午放学前）。
* **多模式配送：**
  * 商家自送（距离较近）。
  * 平台专送（由平台组织的本地配送员）。
  * 顺路代买/代送（结合顺风车模块，村民互助）。
* **运费智能计算：** 基于本地村落距离或按件计费的本地化定价策略。

#### 3. 百商联盟 (商家共赢模块)
* **联盟入驻与认证：** 本地信誉良好的商家受邀或申请加入“百商联盟”。
* **跨店优惠绑定 (异业联盟)：** 核心玩法。例如在A店买菜，可获赠B店买调料的折扣券；在C店修自行车，可获赠D店五金配件抵扣券。
* **积分与权益共享：** 消费者在联盟内任意店铺消费均可累计“同盟积分”，用于兑换实物或抵扣配送费。

#### 4. 熟人顺风车与互助出行 (社交出行模块)
* **强信任准入机制：** **严格限制**。仅支持熟人或同村（如基于村长邀请码、村委白名单认证、已有村民担保认证）。
* **场景化出行发布：** 针对特定场景优化发布流程：如“接送小朋友上下学”、“老人镇上/县城看病接送”、“顺路赶集”。
* **信息互通与安全保障：** 行程仅对同村/同社区人员可见，支持紧急联系人一键呼叫，确保出行绝对安全。

### 2.2 非功能性需求 (Non-Functional Requirements)
* **易用性：** 界面必须极度适老化，大字体、语音播报输入、操作层级扁平化。
* **高可用与低延迟：** 即使在乡镇网络信号不佳的环境下，也能保证核心功能（如顺风车联系、商品浏览）的顺畅使用（考虑增加本地缓存或离线模式）。
* **安全性：** 用户实名认证数据、村落归属数据需严格加密存储；顺风车行程数据严禁外泄。

## 三、 系统架构设计

### 3.1 总体架构蓝图 (Overall Architecture)

系统采用经典的**前后端分离**架构，结合**微服务思想**进行模块划分，以适应未来业务的扩展。考虑到项目前期成本和用户习惯，首选**微信小程序**作为用户触点。

* **表现层 (客户端)：**
  * 用户端微信小程序（商品浏览、下单、顺风车、联盟权益）。
  * 商家端微信小程序（商品管理、订单处理、联盟活动配置）。
  * 管理后台 Web 端（平台运营、审核、数据分析）。
* **接入层 (网关)：** Nginx，负责请求转发、负载均衡、HTTPS 加解密。
* **业务逻辑层 (后端服务)：** 采用 Node.js (如 NestJS 或 Express) 构建核心服务接口。
* **数据持久层：** MySQL (结构化业务数据) + Redis (高频缓存与会话) + OSS (图片/视频附件)。

### 3.2 模块架构图 (Module Architecture)

1. **用户中心 (User Center):** 微信登录、实名认证、村组/网格归属绑定、熟人关系链管理。
2. **商品与商家中心 (Merchant & Product):** 店铺信息、商品分类、价格库、库存展示、评价系统。
3. **交易与调度中心 (Order & Dispatch):** 购物车、预约下单、配送费计算、订单状态流转。
4. **营销与同盟中心 (Marketing & Alliance):** 联盟商家管理、跨店优惠券生成与核销、联合促销规则引擎。
5. **互助出行中心 (Carpool):** 行程发布（车找人/人找车）、基于关系的可见性过滤、路线匹配、油费分摊/红包打赏。
6. **基础服务 (Infrastructure):** 支付服务 (微信支付)、通知服务 (微信订阅消息/短信)、LBS 地理位置服务 (腾讯地图API)。

### 3.3 顺风车模块安全架构专项设计
鉴于此模块的敏感性，必须设计专门的安全屏障：
* **数据隔离：** 行程数据以“村/社区”为逻辑租户进行隔离查询。A村用户无法查看B村的顺风车信息，除非双方有明确的担保关联。
* **准入沙箱：** 新用户注册后默认无权限使用出行功能，必须经过：身份信息校验 -> 本村熟人扫描专属邀请码 -> 平台后台人工复核，三步才能解锁。

## 四、 数据库架构概览 (核心实体关系)

* **User (用户表):** 包含基础信息、实名状态、所属村组 ID。
* **Village_Relation (熟人关系表):** 记录用户之间的邀请、担保关系，构建村落社交图谱。
* **Merchant (商家表):** 店铺名称、地址、经纬度、联盟状态。
* **Product (商品表):** 关联商家，包含名称、价格区间、状态、库存标签。
* **Order (订单表):** 关联用户与商家，包含预约配送时间、配送方式、支付状态。
* **Alliance_Coupon (联盟优惠券表):** 记录发放店铺、可用店铺列表、抵扣规则。
* **Carpool_Trip (行程表):** 关联车主/乘客、出发地/目的地、时间、场景类型(老人看病/小孩上学)、可见范围。

## 五、 后续实施建议

1. **冷启动策略：** 平台初期无需覆盖所有品类。建议从**“新鲜蔬菜”**和**“顺风车”**这两个极高频、痛点最明显的场景切入，先建立本村的“信任飞轮”。
2. **轻资产运营：** 平台主要作为“信息撮合”和“规则制定者”，配送初期可尽量利用商家自有运力或顺风车捎带，降低自建物流成本。
3. **下沉市场调研：** 架构设计已考虑适老化，但在实际UI开发前，务必拿着纸质原型图与老家几位长辈和小卖部老板沟通，验证操作逻辑是否足够简单。
