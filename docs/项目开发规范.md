# 地方特色局域网络物资配送平台 - 项目开发规范文档

## 一、 项目概述与目的

本项目致力于打造一个服务于特定地域（老家乡镇、村落、社区）的综合性本地生活服务平台，名为“乡里乡亲”。

### 1.1 核心目的
- **打破信息壁垒**：解决下沉市场本地商品信息不对称的问题。
- **降低物流成本**：通过“平台集散统配”+“固定班次预约”模式，解决最后三公里配送难题。
- **赋能本地商家**：通过“百商联盟”和村级推客，帮助本地实体商家进行联合营销。
- **解决出行痛点**：建立基于“熟人社会”信任机制的乡村顺风车互助平台，保障老人、儿童安全出行。

### 1.2 目标受众与设计原则
- **目标受众**：乡镇及农村居民（各年龄段，包含大量中老年人）、本地实体商家、兼职配送员/顺风车车主。
- **设计原则**：极简适老化设计、操作层级扁平化、核心流程高容错率。强调“地缘信任”与“信息透明”。

## 二、 架构与技术栈规范

### 2.1 整体架构
采用前后端分离微服务化思想的架构：
- **客户端**：微信小程序（基于 Uni-app），集消费者端与商家端于一体。
- **服务端**：Node.js 体系（优先推荐 NestJS 框架进行后续开发迭代，或现有 Express 架构）。
- **数据持久层**：MySQL 8.0（关系型核心数据） + Redis（缓存、队列）。

### 2.2 前端技术规范 (miniapp)
- **框架**：Uni-app (Vue 3 语法)。
- **样式**：SCSS/SASS，遵循 BEM 命名规范。
- **UI库**：优先使用 uni-ui，复杂组件可引入 uView 或原生实现，避免包体积过大。
- **请求封装**：统一使用封装好的网络请求工具类（如拦截器处理 token、统一错误处理）。

### 2.3 后端技术规范 (backend / platform-server)
- **框架**：NestJS (TypeScript 强类型) 或 Express。建议新业务逐步向 NestJS + TypeScript 迁移。
- **ORM**：TypeORM 或 Sequelize。
- **接口风格**：RESTful API 设计风格。
- **安全**：JWT 无状态鉴权；敏感信息（手机号、身份证、密码）加密存储。

## 三、 目录结构与命名规范

### 3.1 目录结构
项目保持统一的清晰结构，严禁跨模块乱放文件。

```text
delivery-platform/
├── backend/                # 推荐：NestJS 后端服务（基于 TS）
├── platform-server/        # 现有：Express 后端服务（基于 JS）
├── miniapp/                # Uni-app 小程序前端
│   ├── api/                # 接口请求中心
│   ├── components/         # 全局复用组件
│   ├── pages/              # 页面目录 (按业务模块划分文件夹)
│   ├── static/             # 静态资源 (图片、图标)
│   ├── store/              # 状态管理 (Pinia/Vuex)
│   └── utils/              # 工具函数
└── docs/                   # 项目文档中心
```

### 3.2 命名规范

#### 前端命名
- **组件名**：采用 PascalCase（大驼峰），如 `ProductList.vue`。
- **页面文件夹**：采用 kebab-case（短横线），如 `pages/product-detail/`。
- **变量/函数**：采用 camelCase（小驼峰），如 `getUserInfo`，`orderList`。
- **常量**：采用 UPPER_SNAKE_CASE，如 `MAX_UPLOAD_SIZE`。
- **CSS 类名**：采用短横线和小写字母，尽量遵循 BEM，如 `product-card__title--active`。

#### 后端与数据库命名
- **数据库表名**：采用小写及下划线，带模块前缀，如 `sys_user`, `biz_product`, `ord_main`。
- **数据库字段名**：采用小写及下划线，如 `create_time`, `is_certified`。
- **API 路由**：采用全小写和短横线，名词复数，如 `/api/products/:id`。

## 四、 接口与数据交互规范

### 4.1 接口响应统一格式
所有后端 API 必须返回结构化的标准 JSON 数据：
```json
{
  "code": 200,             // 业务状态码：200 成功，4xx 客户端错误，5xx 服务端错误
  "msg": "操作成功",       // 友好的提示信息
  "data": { ... }          // 实际数据承载对象/数组
}
```

### 4.2 错误处理规范
- 前端拦截器应统一拦截 `code !== 200` 的请求，并根据 `msg` 弹出提示。
- Token 过期（如 401 状态码），前端统一拦截并静默刷新或重定向至登录页。

## 五、 Git 提交与分支规范

### 5.1 分支管理
- `main` / `master`：生产环境主分支，保持随时可发布状态。
- `develop`：开发环境主分支，汇总各特性分支的代码。
- `feature/*`：特性分支（如 `feature/carpool`），开发新功能时从 `develop` 检出。
- `bugfix/*`：修复测试环境 bug 分支。
- `hotfix/*`：修复线上紧急 bug 分支，从 `main` 检出。

### 5.2 Commit 提交规范
遵循 Angular 规范，提交格式为：`<type>(<scope>): <subject>`

- `feat`: 新功能 (feature)
- `fix`: 修复 bug
- `docs`: 文档修改
- `style`: 代码格式修改，不影响逻辑
- `refactor`: 重构（非 bug、非新功能）
- `perf`: 性能优化
- `test`: 增加或修改测试用例
- `chore`: 构建过程或辅助工具变动

**示例**：`feat(carpool): 增加顺风车发布行程页面及接口`

## 六、 业务逻辑核心红线

1. **隐私安全红线**：用户身份证、手机号必须加密入库。小程序端展示必须脱敏（如 `138****1234`）。
2. **顺风车安全红线**：顺风车行程数据必须严格以“村落/社区”为维度进行租户级别的数据隔离。跨村用户绝不可见。
3. **资金分账红线**：涉及跨店合单及团长分佣的订单，必须严格记录总单、子单、佣金、运费拆解明细，依托微信支付分账能力确保资金合规，平台不私设资金池截留货款。
4. **性能红线**：首页加载、商品列表加载需控制在 2 秒内。图片上传必须进行压缩处理。
